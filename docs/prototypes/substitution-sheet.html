Substitution Bottom-Sheet

Product Designer spec · Hone v0.5 · 29 April 2026

Interactive Prototype

What this is

The visible surface for the substitutions[] data already in the schema. This is Hone's primary competitive differentiator over Yummly and Supercook — honest, specific swap guidance with a quality rating for each alternative.

Quality pills

Perfect swap perfect_swap
Great swap great_swap
Good swap good_swap / good
Compromise compromise

No traffic-light red. There is no "bad substitute" in the schema — if the result is bad, it doesn't get listed. The pill label communicates quality; the changes text explains the tradeoff.

Ingredient row affordance

Ingredients with substitutions show the ⇌ swap icon in --primary-ink. Ingredients without substitutions show no icon and are not tappable. Whole row is the tap target (min 56dp height satisfies 44dp floor).

Hard-to-find flag

When hard_to_find: true on the original ingredient, a sky-coloured information row appears under the substitution's changes text — not intrusive, but surfaced before the user goes shopping. Uses sky tokens as a neutral info colour (not warning amber, which implies danger).

Design notes

Why a bottom sheet and not an inline dropdown or modal?

Inline dropdowns on long ingredient lists would require the user to scroll back to the right position after the expanded row pushes content down. Modals are disorienting on mobile — they break spatial context. The bottom sheet is the established Android pattern for contextual actions: it keeps the list visible above it (scrim but readable), and dismisses with a familiar swipe-down or backdrop tap. BottomSheetModal from @gorhom/bottom-sheet is the implementation path — it handles safe insets, keyboard avoidance, and fling gestures natively.

Why no traffic-light colours on quality pills?

Red would signal "don't use this" — but every substitution listed is plausible, just with a tradeoff. Calling something red would undermine trust in the whole system. The "Compromise" pill in warm grey with muted text communicates "use with awareness" rather than "avoid". The changes text does the actual communicating. Colour is a supporting signal only.

Why does confirming update the ingredient name in the list?

The user should be able to glance at the ingredient list and immediately know what to buy. If they've swapped beef mince for wagyu, the list must reflect that — otherwise they'll notice the discrepancy mid-shop and have to remember which swap they made. A small "Swapped from: X" caption maintains reversibility without requiring a separate undo flow. The swap icon remains on the row so they can re-open the sheet and change again.

Why is "Keep original" a text button and not a secondary button?

Users will tap into the sheet to browse substitutions without committing. If "Keep original" looked like an equal action to "Confirm", it would create a binary choice framing — "do you swap or not?" The text link reads as dismissal, not a decision, which matches the mental model: browsing is free, confirming is the action.

Accessibility

Sheet has role="dialog" and aria-modal="true". Title has aria-labelledby. Sub-list uses role="radiogroup" with each row as role="radio". Ingredient rows that are tappable carry a TalkBack description: "[Name]. Has substitution options. Double-tap to view." All interactive touch targets are minimum 44dp. The confirm button state is communicated via disabled attribute, not just visual dimming.
