Design rationale
Why the unit goes inside the stepper, not beside it

The number and its unit are one piece of information — "4 burgers" is atomic. Separating them (number inside stepper, unit outside) forces the user's eye to glue two separate elements together. The stacked layout (number above, unit below, within the centre cell) keeps them as a single readable chunk. It also means the label area on the left doesn't have to carry unit context — it stays clean as a single-word verb.

Verb derivation — one field, not two

The verb ("Serves" vs "Makes") is derived automatically: if the unit is "person/people", the verb is "Serves". Everything else uses "Makes". This means the recipe author only sets serving_unit — the UI handles the verb. No risk of mismatched data ("Makes 4 people" from a copy error).

This is a Golden Rule #2 consequence: bread scales by loaf count, not people. Labelling bread as "Serves N" is actively misleading — it implies one loaf feeds N, which isn't how bread works.

Singular / plural — two fields, one fallback

The schema adds serving_unit_singular and serving_unit_plural. Irregular plurals (loaf/loaves) need explicit storage — you can't derive "loaves" from "loaf" without a dictionary. Regular plurals (burger/burgers, cup/cups) can be derived with a simple + "s" fallback, so the plural field is optional.

The engineer should apply the fallback: if serving_unit_plural is absent, display serving_unit_singular + 's'. This keeps seed data minimal for regular-plural units.

Minimum / maximum bounds

Minimum is always 1 — you can't make 0 of anything. The − button disables (opacity 0.28, pointer-events none) at count = 1. Maximum defaults to 8 for people-based recipes and is recipe-defined for unit-based (bread: 4 loaves is a reasonable max for a home kitchen; tortillas: 24; burgers: 12). Defaults if unset: 8 for people, 8 for everything else.

The max isn't about blocking the user — it's about preventing ingredient amounts that become absurd (400g × 20 = 8kg of pasta water). Warn, not error.

The "scales ingredients" hint on bread

Bread is the edge case where users have learned to think of "servings" as people. A loaf doesn't serve 1 person — it serves 8-10 slices. The subtle italic hint "scales ingredients" below the verb on bread/baked-good recipes signals: this control changes how much you make, not who you're feeding. It's a disambiguation line, not an instruction. Muted, 10px italic — present for bread, absent for everything else.

Centre cell sizing

The centre cell grows from 28×32px to 52×40px. The 52px width accommodates "tortillas" (the longest unit label at 8 chars) without text overflow. The 40px height (up from 32px) gives the number and unit enough vertical breathing room to read as two distinct lines rather than a squashed stack. The stepper buttons stay at 32×40px — only the centre zone grows.

'Serves' : 'Makes'; const showHint = verb === 'Makes' && recipe.serving_unit_singular !== 'burger' && recipe.serving_unit_singular !== 'cup'; // Hint text: "scales ingredients" // Only shown for baked-goods / quantity items where // users might misread "Makes 1" as "serves 1 person"

Component changes (recipe/[id].tsx — ServingsSelector or inline)

  1. Centre cell: width: 52, height: 40 (was width: 28, lineHeight: 32). Contains two <Text> nodes: number (15px bold ink) and unit (9px muted). flexDirection: 'column', alignItems: 'center', justifyContent: 'center'.
  2. Left label area: render verb (13px semibold ink-soft). If showHint is true, render a second <Text> below: "scales ingredients" (10px italic muted).
  3. − button: opacity: count === 1 ? 0.28 : 1, pointerEvents: count === 1 ? 'none' : 'auto'.
  4. + button: opacity: count === (recipe.serving_max ?? 8) ? 0.28 : 1, same pointer-events logic.
  5. No other changes — tap logic, state, and ingredient scaling calculations are unchanged.

What does NOT change

Ingredient scaling logic in scale.ts is unchanged — it already operates on count multiples of base_servings. The new fields only affect display labels and stepper bounds. The scaling math is identical.
