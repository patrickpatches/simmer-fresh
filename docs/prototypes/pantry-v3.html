Design Rationale — Pantry v3

Four problems, four decisions

0. Search bar — holistic rethink

The current bar reads as a section heading because it has no visible surface. Text on a dark background without a containing shape is a label, not an input. The fix is a clearly bounded, shaped input field with a warm gold-tint border at idle and a full gold focus ring when active. Alternative 1 is recommended over a sticky top bar because the Pantry screen's primary purpose is browsing and editing an existing list — search is a secondary action. Leading with search (sticky top, Spotify pattern) implies the screen is about finding things, which is wrong for this context.

The rounded-rect shape (14px radius) matches the existing input language in the app (substitution sheet, add-ingredient field). No new interaction language for the user to learn. The gold-tint border at idle is the tell: it's not a neutral grey field, it's part of the Hone palette, and it reads "tap me" without requiring a label to say so.

1. Full-screen search — reverting to inline

The v2 full-screen takeover was the wrong call. Patrick's on-device instinct was correct: when you hide the pantry context (the pills of what you've already added) behind a full-screen search overlay, the user is flying blind. They can't see what they have while deciding what to add next. Inline results fix this architecturally — the header is frozen, the pills row stays visible above the search bar, and the autocomplete list replaces only the ingredient section-list below it.

Cancel dismisses the search and restores the ingredient list. The page never fully hands off to a search context — it borrows the lower half of the screen for autocomplete results while keeping the top half intact. This is the correct pattern for augmentation flows.

The leaf emoji issue (emoji appearing on its own line, separate from the ingredient name) is fixed: the emoji and name are now in the same flex row. No text wrapping between them.

2. Clear all — moved and gated

Two fixes applied, not one. First, the button is moved from the cramped zone directly below the search bar (prime thumb-hit territory) to a trash icon in the title row — a more intentional position that requires deliberate navigation to the top of the screen. Second, tapping it now triggers a confirmation modal that names the exact count: "Clear 12 stocked ingredients." The count matters — a user with 3 ingredients and a user with 40 ingredients have very different stakes. The modal primary button is red and reads "Clear 12 ingredients" (not just "Clear all"), making the consequence concrete. Cancel is the visually dominant choice — it's larger and reads first.

Long-press on pill was considered but rejected: hidden gestures require discovery, and a destructive action should not depend on a gesture the user might never find. A visible icon with a confirmation modal is predictable and matches Android Material 3 conventions for destructive actions.

3. "Getting close" / match banner — explicit CTA

The bare › arrow was doing too much work as an interactive affordance. The new version adds a "See all" pill on the right — a small contained element that reads as a button without requiring the user to infer that the whole card is tappable. The gold color remains throughout (title, pill, arrow inside pill). Sage is removed from the banner — sage means "complete" or "success" in this app's colour language; a set of partial matches is not a success state and shouldn't use that colour.

The title copy is also tightened: "3 recipes you can cook now" is the primary reward signal; "6 more within 1–3 ingredients" is the secondary motivator. These two lines give the user a reason to tap and a sense of what they'll find inside.

What's unchanged

Pill style for added ingredients is preserved. Toggle rows ("Got it? / ✓ Got it") are unchanged. Dark v0.7 tokens throughout. Variant A missing-ingredient chips carry forward from v2. "Photos soon" badge on recipe cards carries through per DECISION-003.

Engineer Handoff — Pantry v3
Implementation notes

1. Search input — replace the existing TextInput wrapper

The current search bar is a View + TextInput with minimal styling. Replace the outer container with the new style:

  • backgroundColor: tokens.surface, borderWidth: 1.5, borderColor: 'rgba(232,184,48,0.22)', borderRadius: 14, paddingHorizontal: 14, paddingVertical: 12.
  • On focus: borderColor: tokens.primary + add a shadowColor: tokens.primary, shadowOpacity: 0.10, shadowRadius: 6, elevation: 0 focus ring.
  • Left icon: magnifying glass, switches from tokens.muted → tokens.primary on focus.
  • Right: clear (×) button when input non-empty; nothing when empty.
  • Cancel button: appears to the right of the input box on focus (not inside it). Tapping dismisses focus, clears the search state, restores the ingredient list. onPress: () => { setSearch(''); Keyboard.dismiss(); }

2. Inline autocomplete — replace full-screen IngredientSearchOverlay

The IngredientSearchOverlay modal built in v2 should be removed. Inline results are implemented within the existing SectionList:

  • Add isSearchActive state (true when addName.length > 0 and input is focused).
  • When isSearchActive: the SectionList's sections prop switches from the pantry category sections to the autocomplete results sections (same SectionList, different data source). Animate with a short fade: useAnimatedStyle + withTiming(opacity, 150ms).
  • Autocomplete result items: emoji and name in the same flex row — emoji is a Text at 14sp with marginRight: 8, name is a Text at 14sp in the same View flexDirection="row". This fixes the leaf-emoji separate-line bug.
  • Already-added items: dimmed (opacity: 0.45) with a "✓ Added" label in tokens.sage on the right. Not tappable (or tapping does nothing).
  • The frozen header (pills row + search bar) stays mounted in ListHeaderComponent regardless of isSearchActive. Only the section data changes.

3. Clear all — relocate and gate

  • Remove the current "Clear all" button from below the search bar.
  • Add a trash icon button (Icon name="trash-2" or similar, 16sp, tokens.muted) to the title row, right-aligned. Wrap in a Pressable with hitSlop={12}.
  • On press: show a Modal confirmation dialog (not an Alert — Alert can't be styled to match dark tokens). Dialog content:
    • Title: "Clear your pantry?" in fonts.display.
    • Body: "This removes all {haveCount} stocked ingredients. Your recipe matches will reset." — include the count, make it concrete.
    • Primary button: backgroundColor: '#E05252' (danger red), label "Clear {haveCount} ingredients".
    • Cancel button: backgroundColor: tokens.surface2, label "Keep my pantry" (positive framing — the cancel is the safe choice).
  • On confirm: clearAllPantryItems(db) → setPantryItems([]). Haptic: Haptics.notificationAsync(ImpactFeedbackType.Warning).

4. Match banner — update copy and CTA

  • Title: "{canMakeCount} recipe{canMakeCount !== 1 ? 's' : ''} you can cook now" in fonts.display, tokens.primary (gold).
  • Sub-line: "{topMatches.length - canMakeCount} more within 1–3 ingredients".
  • Replace bare › with a "See all" pill: backgroundColor: tokens.primaryLight, borderRadius: 999, paddingHorizontal: 10, paddingVertical: 4. Label "See all" in tokens.primary 11sp 700 weight, followed by a small right-arrow SVG.
  • Remove sage colour from this card entirely — sage means complete/success elsewhere in the app.

5. Files to touch

  • mobile/app/(tabs)/pantry.tsx — all four changes above.
  • mobile/src/components/IngredientSearchOverlay.tsx — delete this file (inline supersedes it).
  • No new tokens. No schema changes. No new packages.
